## Copywriting Contract

### Context Menu Items

| Element | Copy | Source |
|---------|------|--------|
| Parent menu item title | `Move to Workspace` | D-13 (CONTEXT.md) |
| Submenu entry — inactive workspace | `{name} ({n} tabs)` e.g. `Work (12 tabs)` | D-10 (CONTEXT.md) |
| Submenu entry — workspace active in another window | `{name} ({n} tabs) [open]` e.g. `Work (12 tabs) [open]` | D-11 (CONTEXT.md) + Research recommendation |
| Submenu entry — 1 tab | `{name} (1 tab)` — singular, not "1 tabs" | Grammar default |

### Interaction Feedback

| Element | Copy | Rationale |
|---------|------|-----------|
| Error notification — move failed | No popup toast or banner. Log to console: `[Workspaces] Move error: {e.message}`. Silent failure from user perspective unless a future notification phase adds UI. | Claude's Discretion — no notification UI in scope for this phase |
| Empty submenu state | Not possible: submenu only appears when ≥ 1 workspace other than the active one exists. If only one workspace exists, "Move to Workspace" parent item is still shown but has no children — Firefox will hide or gray it out natively. No copy needed. | Inferred from D-15 (active workspace excluded) |

### No Destructive Actions in This Phase

Moving tabs is a non-destructive, reversible operation (tabs move to another workspace, nothing is deleted). No confirmation dialog is required.

**Source:** CONTEXT.md D-08 (empty source workspace is allowed, no auto-delete), D-09 (atomic with rollback). No confirmation pattern needed.

## Interaction Contract

These are the behavior rules the executor must implement. Not styling — wiring.

### Submenu Population

- Parent item `"Move to Workspace"` is created once in `browser.runtime.onInstalled`. It persists.
- Children are rebuilt on every `menus.onShown` event (not cached, not incrementally updated).
- Children are filtered: active workspace for the current window is excluded (D-15).
- Children are sorted by `lastUsedAt` descending, falling back to `createdAt` if `lastUsedAt` is absent (D-12).
- Each child label follows the copywriting contract above.
- `menus.refresh()` is called after rebuilding children to commit changes before display.

### Multi-Tab Selection Logic (MENU-04)

- On `menus.onClicked`: if `tab.highlighted === true`, query all highlighted tabs in the window with `browser.tabs.query({ windowId: tab.windowId, highlighted: true })` and move all of them.
- If `tab.highlighted === false` (user right-clicked a non-selected tab), move only that one tab.
- This prevents the always-highlighted active tab from being accidentally included.

### "Active in Another Window" Indicator (D-11)

- Determined at menu-open time: a workspace is "open" if its `id` appears as any value in `getWindowMap()`.
- Label format: `{name} ({n} tabs) [open]` — plain text suffix, no icon (menu item icons are possible via the `icons` property on submenu children, but text suffix is simpler and sufficient).
- Clicking an `[open]` workspace triggers the cross-window move path: `browser.tabs.move()` + `browser.windows.update(..., { focused: true })`.

### Tab Count in Label

- Tab count shown in submenu labels reflects `ws.tabs.length` from storage (the stored tab list), not the live open-tab count.
- For a workspace active in another window, the live count may differ from storage. This is acceptable — the stored count is used for simplicity. Implementation note: for cross-window targets, the label may be slightly stale; this is not a blocking issue.

### `lastUsedAt` Field

- Add `lastUsedAt: Date.now()` to workspace objects when a workspace becomes the active destination (in `switchWorkspace()` and in the new `moveTabsToWorkspace()`).
- The field must round-trip through `serializeToSyncItems` / `assembleFromSync` in `sync.js`.
- Workspaces without `lastUsedAt` fall back to `createdAt` for sorting.
